#6- Ruby's Got Class

February 22, 2016

Ruby's my first. She's pretty and well-spoken. Most importantly, she has class. But, the badboy in me craves more danger, more consequences. Oooh...consequences. My friends at Dev Bootcamp eventually introduce me to a fox named JS, or JavaScript(no, not the actress). Like any interested man, I read about her in a book written by a guy who's been following her for years.

I've been with JS for a week now. She speaks a different language than Ruby, but it isn't too difficult understanding her because, like Ruby, JS speaks with objects. Every action and characteristic is an object for them. So, what's so different between them besides syntax? JS can sometimes be a little too liberal for my own good. For example:

console.log("8" - 2)
   -> 6
console.log("8" + 2)
   -> 82
While Ruby just wouldn't let me add or subtract a string and integer, JavaScript automatically converts the data types to try to make them work. In our first case, "8" is converted to the integer 8 to make 8 - 2 equate to 6. But, in the next example, the 2 is converted into "2," making "8" + "2" concatenate to "82." So, I'm left asking...

Where's Her Class?

Ruby comes from a place where Class defines how one behaves. Class is what's called a 'metaobject,' or object that defines objects. Remember that both Ruby and JS speak in objects. Where JS comes from, however, they don't have Class titles. This slight difference in organization represents why JS has more freedom but also more problems, and why Ruby's structure can bring convenience.

Different Name, Same-ish Game

Ruby's metaobject is the Class object. New instances call their methods through Ruby's method of inheritence, which allows instances to 'call' on the methods of the class rather than having to create their own versions of the methods. JavaScript has two metaobjects: The 'constructor' function object and the 'prototype' object. Their purposes individually align with the general purpose of a metaobject. But, to emulate a Ruby class, both the constructor function and the prototype property are required. The problem with defining methods in a constructor function alone is the inefficiency caused by each instance's redeclaration of the constructor's methods. The prototype property allows for inheritence directly from the constructor rather than redeclaration in each instance. Because methods are inherited in this way, the prototype property can also define what is basically a class method from outside of the constructor.
